# Лекция №3

## Оперативная память, кодировки и стек

### 1. Абстрактная модель оперативной памяти

* **Бит** — минимальная единица хранения данных.
* Процессор не работает с отдельными битами: минимальной адресуемой единицей является **байт** (8 бит).
* Например, тип `bool`, несмотря на логическое значение «истина/ложь», занимает **1 байт** памяти.

#### Почему 8-битные ячейки?

* Исторически сложилось из-за стандарта **ASCII**.
* При проектировании первых компьютеров выяснилось, что использовать 8-битные ячейки технически проще, чем 6- или 7-битные.
* В итоге **все производители пришли к стандарту 8-битного байта**.

### 2. Размер оперативной памяти и адресация

Максимальный объём адресуемой памяти определяется разрядностью адресного регистра:

\[ \text{Максимальный объём памяти} = 2^{\text{разрядность регистра\}} \text{ байт} ]

Примеры:

* 8-битный регистр → 256 байт
* 10-битный регистр → 1024 байта (1 КБ)
* 32-битный регистр → 4 ГБ

### 3. Хранение данных в памяти

#### Порядок байтов (Endianness)

* При хранении многобайтовых значений (например, `0x04D8`) возможны два подхода:
  * **Big-endian**: старший байт по младшему адресу.
  * **Little-endian**: младший байт по младшему адресу (используется в большинстве современных архитектур, например, x86).
* Проблема возникает при передаче данных между системами с разным порядком байтов.

#### Выравнивание данных

* Процессоры эффективнее работают с данными, выровненными по границам **машинного слова** (например, 4 байта на 32-битной системе).
* Если данные не выровнены, процессору приходится выполнять дополнительные операции чтения/записи, что снижает производительность.

### 4. Типы данных

#### Примитивные (значимые) типы

* Имеют фиксированный размер (например, `int`, `char`, `bool`).
* Занимают строго определённое количество байт.
* Неиспользуемые биты дополняются нулями.

#### Ссылочные типы

* Включают указатели, массивы, структуры, объекты.
* В памяти хранится **адрес** начала данных.
* Для доступа к элементу массива используется формула:\
  `адрес = база + индекс × размер_элемента`.

#### Управление памятью

* **Ручное**: программист сам выделяет и освобождает память (например, в C/C++).
* **Автоматическое**: используется **сборщик мусора** (garbage collector), как в Java, C#, Python.

### 5. Кодировки символов

#### ASCII

* Первый стандарт кодировки (1960-е).
* Использует **7 бит** → 128 символов.
* Из них 31 — управляющие (не отображаются).
* Подходит только для английского языка.

#### Расширенный ASCII и кодовые страницы

* 8-й бит позволил добавить ещё 128 символов.
* Разные страны создавали свои **кодовые страницы** (например, CP1251 для кириллицы).
* Проблема: текст, созданный в одной кодовой странице, отображался некорректно в другой.

#### Unicode

* Универсальный стандарт, призванный охватить все письменности мира.
* На момент 2020-х содержит более **140 000 символов**, включая эмодзи.
* Совместим с ASCII: первые 128 кодов идентичны.

#### Кодировки на основе Unicode

**UTF-8**

* Переменная длина: от **1 до 4 байт** на символ.
* Преимущества:
  * Полная совместимость с ASCII.
  * Эффективное использование памяти для латинских текстов.
* Структура:
  * 1 байт: символы U+0000–U+007F (ASCII)
  * 2 байта: U+0080–U+07FF
  * 3 байта: U+0800–U+FFFF
  * 4 байта: U+10000–U+10FFFF
* Не требует BOM (Byte Order Mark) — порядок байтов однозначен.

**UTF-16**

* Использует **2 или 4 байта** на символ.
* Символы из основной плоскости (U+0000–U+FFFF) кодируются 2 байтами.
* Символы за пределами — **суррогатными парами** (по 2×16 бит).
* Используется в Java, JavaScript, C#.
* Быстрее UTF-8 при обработке, но менее экономичен по памяти для латиницы.

**UTF-32**

* Фиксированная длина: **4 байта** на символ.
* Прост в обработке, но неэффективен по памяти.
* Используется, например, в Python (внутреннее представление строк в некоторых версиях).

### 6. Строки в языках программирования

* Строка — это **массив символов**.
* Реализация зависит от языка:
  * **C**: строка завершается нулевым байтом (`\0`). Длина вычисляется при каждом обращении → медленно.
  * **Java / C# / Python**: строки **неизменяемы** (immutable). Любая модификация создаёт новую строку.
* Это влияет на производительность и потребление памяти.

### 7. Стек (Stack)

#### Общие сведения

* Стек — область памяти для хранения **временных данных**: параметров функций, локальных переменных, адресов возврата.
* Работает по принципу **LIFO** (Last In — First Out).
* В отличие от **кучи** (heap), где хранятся долгоживущие объекты, стек автоматически управляется процессором.

#### Адресация стека

* В большинстве архитектур стек **растёт в сторону уменьшения адресов**.
* Это сделано для предотвращения пересечения с другими сегментами памяти (например, кодом или кучей).

#### Регистры, связанные со стеком

* **SP (Stack Pointer)** — указывает на вершину стека.
* **BP (Base Pointer)** — указывает на начало текущего **стекового фрейма** (frame). Используется для доступа к параметрам и локальным переменным.

#### Операции со стеком

* `push X` — помещает значение `X` в стек:\
  `SP = SP - размер(X)`, затем запись по новому адресу.
* `pop X` — извлекает значение из стека:\
  чтение по `SP`, затем `SP = SP + размер`.

### 8. Стек и вызов функций

#### Стековый фрейм

* При вызове функции создаётся **фрейм** — блок памяти в стеке, содержащий:
  * Адрес возврата
  * Сохранённое значение BP
  * Параметры
  * Локальные переменные

#### Пролог функции (начало)

```asm
push bp        ; сохраняем старый BP
mov bp, sp     ; BP теперь указывает на начало фрейма
sub sp, N      ; выделяем N байт под локальные переменные
```

#### Эпилог функции (завершение)

```asm
mov sp, bp     ; восстанавливаем SP
pop bp         ; восстанавливаем старый BP
ret            ; возврат по адресу из стека
```

#### Передача параметров

* Параметры обычно передаются **в обратном порядке** (справа налево).
*   Пример вызова `f(1, 2, 3)`:

    ```asm
    push 3
    push 2
    push 1
    call f
    ```
* Внутри функции:
  * `[bp + 4]` — первый параметр
  * `[bp + 6]` — второй (если параметры 2-байтные) и т.д.

> **Важно**: доступ к параметрам и переменным следует осуществлять **через BP**, а не SP, так как SP может меняться во время выполнения функции.

#### Способы передачи аргументов

1. **Через стек** — универсально, но медленнее.
2. **Через регистры** — быстрее, но ограничено количеством регистров.
3. **Смешанный подход** — используется в современных ABI (Application Binary Interface).
